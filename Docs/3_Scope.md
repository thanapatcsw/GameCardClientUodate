# 3. ขอบเขตของโครงการและการพัฒนา (Scope)

โครงงานนี้มุ่งเน้นการออกแบบและพัฒนาเกมการ์ดดิจิทัลเชิงศึกษาที่รันอยู่บนอุปกรณ์พกพา รวมถึงระบบการซิงโครไนซ์สถานะผู้เล่นหลายคนแบบเรียลไทม์ และระบบบริหารจัดการฐานข้อมูลคลาวด์หลังบ้าน โดยมีขอบเขตการทำงานจำแนกตามระบบย่อย 8 ระบบดังต่อไปนี้:

- 3.1 ระบบจัดการผู้ใช้งานและการยืนยันตัวตน (User Authentication & Web-based Registration)
  - ระบบลงทะเบียนสมาชิกใหม่ผ่านเว็บ (Web-based Registration): พัฒนาหน้าเว็บลงทะเบียน (index.html) แยกส่วนจากตัวเกมหลักด้วย HTML5, CSS (Vanilla CSS) และ JavaScript โดยรับข้อมูลชื่อผู้ใช้ (Username), อีเมล (Email) และรหัสผ่าน (Password) ของผู้เล่น จากนั้นส่งข้อมูลเชื่อมต่อไปยังบริการ Supabase Authentication เพื่อสมัครและบันทึกบัญชีลงทะเบียนใหม่
  - ระบบล็อกอินและเข้าสู่ระบบในแอปพลิเคชัน (Application Login Scene): พัฒนาฉากการเข้าสู่ระบบ (LoginScece.unity) บนเอนจิน Unity เพื่อรองรับผู้เล่นในการกรอกอีเมลและรหัสผ่านเพื่อยืนยันตัวตนกับ Supabase Auth เมื่อการเข้าสู่ระบบผ่านการสิทธิ์เรียบร้อย ระบบจะดึงโปรไฟล์ส่วนตัวของผู้เล่นพร้อมส่งผู้ใช้งานไปยังหน้าต่างเมนูหลักของเกมโดยอัตโนมัติ
  - ระบบแก้ไขข้อมูลส่วนตัว (Profile Customization): ผู้เล่นสามารถเข้าไปทำการแก้ไขชื่อแสดงผล (Display Name) หรือสลับการติดตั้งกรอบรูปโปรไฟล์อวตาร์ (Avatar Frames) ที่ได้รับจากการปลดล็อกผ่านระบบร้านค้าในภายหลังได้จากส่วนการตั้งค่าภายในแอปพลิเคชัน

- 3.2 ส่วนผู้ดูแลระบบสำหรับการจัดการคลังคำถาม (Web Admin Interface)
  - การบริหารคลังคำถามผ่านระบบคลาวด์ (Database Management Panel): ระบบสนับสนุนการทำงานของผู้ดูแลระบบ (Admin) ในการเข้าถึงส่วนควบคุมของฐานข้อมูล Supabase เพื่อดำเนินการเพิ่ม (Create), เรียกดู (Read), แก้ไข (Update) และลบ (Delete) ข้อคำถามเชิงความรู้คอมพิวเตอร์ ซึ่งโครงสร้างข้อมูลประกอบด้วย โจทย์คำถาม, ตัวเลือกตอบ 4 ตัวเลือก และดัชนีชี้เฉลยคำตอบที่ถูกต้อง
  - การอัปเดตแบบเรียลไทม์ (Dynamic Update Integration): ทุกการปรับแต่ง จัดการ หรือเปลี่ยนแปลงคลังข้อมูลคำถามบนหลังบ้านของ Supabase จะถูกซิงค์เพื่อนำไปใช้แสดงผลในการเล่นของตัวเกมบนอุปกรณ์ผู้ใช้งานโดยอัตโนมัติ

- 3.3 ระบบห้องพักและการจับคู่ผู้เล่น (Matchmaking & Private Lobby System)
  - โหมดการเล่นคนเดียว (Single Player): รองรับระบบการทำงานที่เปิดโอกาสให้ผู้เล่นสามารถสร้างห้องเล่นส่วนตัวกับระบบผู้เล่นคอมพิวเตอร์ (Bot) จนครบตามจำนวนช่องว่างที่เหลือภายในห้อง
  - โหมดการสร้างห้องส่วนตัว (Private Lobby): ผู้เล่นสามารถเลือกสร้างห้องเล่นเกมส่วนตัว และระบบจะทำการสร้างรหัสห้องพัก (Room Code) เพื่อส่งต่อให้ผู้เล่นอื่นพิมพ์ป้อนรหัสผ่าน เพื่อเข้าร่วมกลุ่มเล่นร่วมกันในห้องเดียวกัน (รองรับจำนวน 2-4 คน)
  - โหมดจับคู่เล่นอัตโนมัติ (Auto Matchmaking Queue): พัฒนาสถาปัตยกรรมคิวการรอคอยและการจับคู่แบบเรียลไทม์บนฐานข้อมูลหลังบ้าน โดยอาศัยฟังก์ชันระดับฐานข้อมูล (Database Functions) ได้แก่ claim_matchmaking_room และ cancel_matchmaking_search ในการบันทึกสถานะผู้เล่นเข้าตาราง matchmaking_queue และจัดสรรกลุ่มห้องจริงเข้าตาราง rooms บนระบบ Supabase
  - การเชื่อมโยงระบบเครือข่ายเรียลไทม์: ประยุกต์ใช้แพลตฟอร์ม Photon Fusion (Shared Mode) สำหรับการรับส่งข้อมูลโครงสร้างระบบเครือข่ายเพื่อทำการซิงโครไนซ์สถานะของเกม (Game States), ทรัพยากรกองกลาง, หน้าตักทรัพยากรของผู้เล่น, การตัดสินรอบควิซ และการสลับลำดับเทิร์นการเล่นแบบเรียลไทม์สำหรับกลุ่มผู้เล่นจริง

- 3.4 ระบบการเล่นหลักของเกมการ์ด (Core Gameplay Loop)
  - ธีมการนำเสนอและสภาพแวดล้อม: การดำเนินเนื้อเรื่องหลักของบอร์ดเกมเกิดขึ้นภายใต้เรื่องราวของวงการเทคโนโลยีและการแข่งขันแฮกกาธอน ประกอบด้วยฉากแผนที่หลัก 2 ฉากคือ "Startup City" และ "The Great Hackathon"
  - เป้าหมายการเอาชนะ (Win Condition): กำหนดให้ผู้เล่นทุกคนแข่งขันสะสมแต้มระบบ (Stars) จากการพัฒนาการ์ดโมดูลซอฟต์แวร์ให้ครบ 20 แต้ม (Stars) เป็นคนแรก เมื่อมีผู้เล่นทำคะแนนถึงเกณฑ์แล้ว ระบบจะดำเนินเกมต่อไปให้ครบจบรอบนั้น (เพื่อให้ผู้เล่นทุกคนในรอบนั้นมีจำนวนเทิร์นที่เท่ากันทั้งหมด) ก่อนเปรียบเทียบแต้มสะสมและประกาศชัยชนะ
  - เหรียญทรัพยากรระบบคอมพิวเตอร์ (System Resources): ประกอบด้วยทรัพยากรจำลองในรูปเหรียญ 5 ประเภทหลัก ได้แก่ CPU, RAM, Network, Storage และ Security
  - การ์ดโมดูลซอฟต์แวร์ 3 ระดับ (Software Module Tiers):
    - Tier 1 (ระดับพื้นฐาน): มูลค่าคะแนน 0–1 แต้ม (Stars) มีต้นทุนเหรียญทรัพยากรที่ต้องจ่ายในการพัฒนาต่ำ
    - Tier 2 (ระดับระบบย่อย): มูลค่าคะแนน 1–3 แต้ม (Stars) มีต้นทุนเหรียญทรัพยากรปานกลาง
    - Tier 3 (ระดับองค์กร): มูลค่าคะแนน 3–5 แต้ม (Stars) (สอดคล้องกับ TOR ข้อ 3.2.2 ที่ระบุแต้มการ์ดระดับ 3 มีดาวระหว่าง 3-5 แต้ม) มีต้นทุนทรัพยากรสูง
  - การเลือกกระทำในตาเดิน (Turn Actions): เมื่อถึงคิวเทิร์นการเล่นของตนเอง ผู้เล่นสามารถเลือกทำกิจกรรมหลักได้เพียง 1 อย่างจาก 3 รูปแบบต่อไปนี้:
    - 1. การรวบรวมทรัพยากร (Collect Resources): เลือกหยิบเหรียญทรัพยากรคอมพิวเตอร์ (CPU, RAM, Network, Storage, Security) จากกองกลางตามข้อกำหนดกติกา (หยิบเหรียญต่างกัน 3 สี หรือ หยิบเหรียญสีเดียวกัน 2 เหรียญหากมีจำนวนเหรียญในกองกลางเหลือมากกว่าเงื่อนไขขั้นต่ำ)
    - 2. การพัฒนาโมดูล (ซื้อการ์ด - Purchase Card): จ่ายทรัพยากรตามต้นทุนที่ระบุหน้าการ์ด เพื่อนำการ์ดเข้าพอร์ตการพัฒนาของตนเอง ส่งผลให้ได้รับแต้มคะแนนสะสม (Stars) และได้รับโบนัสลดหย่อนทรัพยากรสีนั้นๆ (ส่วนลดถาวร) สำหรับใช้ลดต้นทุนในการพัฒนาการ์ดใบถัดไปในรอบต่อไป
    - 3. การจองโมดูล (Reserve Card): เลือกจองการ์ดโมดูลบนบอร์ดชั่วคราวขึ้นมาเก็บไว้บนมือของตนเอง (เก็บสะสมในมือได้ไม่เกิน 3 ใบ) เพื่อตัดโอกาสหรือสกัดผู้เล่นอื่น พร้อมรับเหรียญทรัพยากรพิเศษ (Wildcard) 1 หน่วยเพื่อใช้ทดแทนเหรียญทรัพยากรสีใดๆ ก็ได้

- 3.5 ระบบผู้เล่นคอมพิวเตอร์ (Rule-Based Bot System)
  - พฤติกรรมการตอบสนองรอบควิซแบบเลียนแบบมนุษย์: บอทได้รับการออกแบบให้มีพฤติกรรมหน่วงเวลาสุ่มตอบคำถาม โดยระบบจะสุ่มตอบในช่วง 5 วินาทีสุดท้าย จากกรอบเวลากลาง 20 วินาที และกำหนดให้บอทตอบคำถามถูกต้องเสมอ 100% เพื่อกระตุ้นให้ผู้เล่นมนุษย์ชิงจังหวะเอาชนะลำดับเทิร์นด้วยการตอบคำถามให้รวดเร็วยิ่งขึ้น
  - ตรรกะลำดับการตัดสินใจ (4-Level Priority System): บอทจำลองจะสแกนสถานะของกระดานเกมและหน้าตักทรัพยากรที่มีอยู่ จากนั้นจะพิจารณาทำกิจกรรมที่ดีที่สุดตามเกณฑ์ลำดับความสำคัญ 4 ระดับดังนี้:
    - ลำดับที่ 1 (ซื้อการ์ดเพื่อชนะเกม): ตรวจสอบการ์ดบนกระดานและบนมือทั้งหมด หากมีใบใดที่บอทจ่ายซื้อได้สำเร็จแล้วส่งผลให้แต้มดาวรวมสะสมเท่ากับหรือเกิน 20 แต้ม บอทจะทำการซื้อการ์ดใบนั้นและสิ้นสุดรอบเกมทันที
    - ลำดับที่ 2 (ซื้อการ์ดที่ดีที่สุด/คุ้มสุด): ในกรณีที่ยังไม่สามารถปิดเกมได้ บอทจะคัดเลือกการ์ดที่จ่ายซื้อได้และให้ค่าแต้มคะแนนดาวสูงที่สุดก่อน หากมีทางเลือกที่ให้คะแนนดาวเท่ากัน บอทจะตัดสินใจซื้อการ์ดที่มีราคาถูกที่สุด (ใช้ทรัพยากรรวมน้อยที่สุด) เพื่อความคุ้มค่า
    - ลำดับที่ 3 (จองการ์ดตัดหน้าเพื่อประหยัดทรัพยากร): หากเหรียญที่มีไม่เพียงพอในการซื้อการ์ดใบใดเลย บอทจะทำการตรวจสอบหาการ์ดระดับสูง (Tier 3) ที่ตนเองต้องการนำมาพัฒนาและยังขาดแคลนเหรียญทรัพยากรอีกเพียงเล็กน้อย (1-2 เหรียญ) บอทจะพิจารณาทำกิจกรรม "จองการ์ด" ใบนั้นขึ้นมือ เพื่อล็อกเป้าหมายและรับเหรียญพิเศษ (Wildcard)
    - ลำดับที่ 4 (หยิบเหรียญทรัพยากรเพื่อเตรียมพร้อม): หากไม่สามารถซื้อหรือจองการ์ดตามเงื่อนไขข้อ 1-3 ได้ บอทจะทำกิจกรรมสแกนหาการ์ดโมดูลที่ต้องการพัฒนาสูงสุด และเลือกหยิบเหรียญทรัพยากรที่ยังขาดแคลนมาจากกองกลางเพื่อสะสม

- 3.6 ระบบทดสอบความรู้และควิซ (Quiz System & Daily Quiz)
  - ระบบควิซประเมินความรู้คอมพิวเตอร์ระหว่างเล่นเกม (Interventional Quiz): ระบบจะเรียกเปิดหน้าต่างคำถามทดสอบเชิงการศึกษาให้กับผู้เล่นทุกคนขึ้นมาตอบพร้อมกันทุก ๆ 3 รอบการเล่น (ก่อนสลับไปดำเนินเทิร์นในรอบใหม่) ผู้เล่นที่เลือกตอบคำตอบที่ถูกต้องและทำสถิติเวลาตอบเร็วที่สุดจะเป็นผู้ได้รับสิทธิ์เริ่มเล่นคนแรกในเทิร์นรอบถัดไป พร้อมได้รับเหรียญคะแนนตอบคำถามพิเศษเพิ่มเติม 1 เหรียญ
  - ควิซประจำวันเพื่อการประเมินศึกษาแบบเดี่ยว (Daily Quiz Scene): พัฒนาฉากจำลอง DailyQuizScene.unity แยกจากการแข่งขัน ให้ผู้เล่นทั่วไปสามารถเข้าทดสอบความรู้คอมพิวเตอร์แบบออฟไลน์ได้วันละ 1 ครั้ง เพื่อทบทวนวิชาการและสะสมแต้มเพชร (Gems) ในการนำมาแลกเปลี่ยนสกิน

- 3.7 ระบบร้านค้าและการปรับแต่งโปรไฟล์ (Avatar Shop & Customization)
  - ระบบร้านค้าแลกเปลี่ยนด้วยระบบเหรียญรางวัล (Store Scene): พัฒนาฉากหน้าร้านค้าแลกเปลี่ยน (StoreScene.unity) รองรับผู้เล่นในการนำเพชร (Gems) ที่ได้จากการพิชิตเควสตอบควิซประจำวันหรือผลชัยชนะสะสม มาเลือกซื้อหรือปลดล็อกสกินกรอบรูปโปรไฟล์สไตล์ต่างๆ (Avatar Frames)
  - ระบบปรับปรุงอวตาร์โปรไฟล์: รองรับผู้เล่นในการเปิดใช้กรอบรูปโปรไฟล์พิเศษที่ได้ปลดล็อกแล้วเพื่อนำมาแสดงในสถิติโปรไฟล์ และแสดงประดับในตารางระบบจับคู่และจัดอันดับ

- 3.8 ระบบจัดอันดับและสถิติประวัติการเล่น (Leaderboard & Match History)
  - ตารางผู้นำและระบบจัดอันดับเรียลไทม์ (Leaderboard): ระบบพัฒนาให้ทำการจัดสรรแสดงผลอันดับคะแนนผู้เล่นเรียงลำดับจากค่า MMR (Matchmaking Rating) จากสูงสุดลงมา โดยการคัดกรองข้อมูลผู้ใช้งานจากตาราง player_profiles ในฐานข้อมูล Supabase นำมาจัดอันดับและแสดงผลลัพธ์ในหน้าจอจัดอันดับ (RankScene.unity)
  - ระบบจัดเก็บประวัติการแข่งขัน (Match History & Statistics): ระบบหลังบ้านทำหน้าที่คอยอัปเดตสถิติจำนวนตาที่เล่นชนะ (Wins) และจำนวนตาที่แพ้ (Losses) ของแต่ละบัญชีผู้ใช้งานลงฐานข้อมูลอย่างเสถียรหลังจากการสรุปผลชัยชนะในแต่ละเกมกระดาน
