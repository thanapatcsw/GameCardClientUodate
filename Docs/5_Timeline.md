# 5. วิธีการดำเนินงานและการออกแบบระบบ

การพัฒนาระบบเกมการ์ดเสริมสร้างแนวคิดระบบสารสนเทศ ได้ดำเนินการตามระเบียบวิธีวิจัยและขั้นตอนทางวิศวกรรมซอฟต์แวร์ (Software Engineering) เพื่อให้เกิดความสมบูรณ์ในการทำงานประสานกันระหว่างฝั่งผู้ใช้งานบนอุปกรณ์พกพา เซิร์ฟเวอร์แบบเรียลไทม์ และระบบบริหารคลังข้อมูลบนระบบคิวคลาวด์หลังบ้าน โดยขั้นตอนการดำเนินงานและการออกแบบส่วนประกอบต่างๆ มีรายละเอียดดังต่อไปนี้:

- 5.1 ระเบียบวิธีวิจัยและการดำเนินงาน (Research Methodology & Timeline)
  - 5.1.1 ขั้นตอนและกระบวนการทำงาน
    - การศึกษาและเก็บข้อมูล: ศึกษารูปแบบหลักสูตรวิทยาการคอมพิวเตอร์และสถาปัตยกรรมระบบสารสนเทศเพื่อระบุประเภทคลังข้อสอบ 5 หมวดหลัก (CPU, RAM, Storage, Network, Security) พร้อมทั้งวิเคราะห์ฟีเจอร์เกมแบบจับคู่ออนไลน์และการเล่นเกมแบบมีปฏิสัมพันธ์ (Interactive Multi-player Board Game)
    - การวิเคราะห์ระบบและสรุปคุณสมบัติ: กำหนดหน้าที่การทำงานของตัวระบบเกมตามแบบสัญญารับเหมางาน (Terms of Reference - TOR) และโครงร่างความต้องการเชิงหน้าที่ (Functional Requirements)
    - การออกแบบสถาปัตยกรรมระบบและฐานข้อมูล: กำหนดโครงสร้างความสัมพันธ์และพจนานุกรมข้อมูล (Data Dictionary) ในระบบฐานข้อมูลคลาวด์ (Cloud Database) และออกแบบแผนภาพระดับระบบ (Data Flow Diagram และ Core Gameplay Flowchart)
    - การพัฒนาระบบและติดตั้ง: พัฒนาระบบไคลเอนต์ด้วย Unity Engine และ Photon Fusion 2 ร่วมกับการทำงานฐานข้อมูลเรียลไทม์ผ่าน RESTful API ของ Supabase และสร้างอินเทอร์เฟซสมัครสมาชิกสำหรับผู้เรียนผ่านเว็บ
    - การทดสอบและประเมินผลระบบ: ดำเนินการประเมินตรรกะการตัดสินใจของระบบบอทอัตโนมัติ (Bot decision windows) และการจัดตารางเวลาควิซ รวมถึงประเมินประสิทธิภาพการไหลของข้อมูลคิวจับคู่ และนำระบบไปทดสอบความพึงพอใจการใช้งานของกลุ่มตัวอย่างผู้ใช้จริง
  - 5.1.2 แผนการปฏิบัติงานโครงงาน (Gantt Chart Timeline)
    - โครงงานนี้ดำเนินการในกรอบเวลารวม 4 เดือน (ตั้งแต่เดือนพฤศจิกายน พ.ศ. 2568 ถึงเดือนกุมภาพันธ์ พ.ศ. 2569) ครอบคลุมทั้งขั้นตอนการเตรียมงาน ออกแบบ พัฒนา ติดตั้ง เขียนเอกสารรายงาน และส่งมอบงานจริง
    - [วางตารางที่ 1 แผนการดำเนินงานและแผนภูมิแกนต์ (Gantt Chart) ตรงนี้]

- 5.2 การออกแบบระบบฐานข้อมูลและพจนานุกรมข้อมูล (Database Design & Data Dictionary)
  - ระบบฐานข้อมูลเก็บรักษาและประมวลผลข้อมูลผู้เล่น ข้อมูลห้องออนไลน์ และข้อมูลสถานะคิวการจับคู่เรียลไทม์บนคลาวด์หลังบ้าน Supabase (PostgreSQL) เพื่อป้องกันปัญหาสภาวะการแข่งขันแย่งทรัพยากร (Race Conditions) ในระบบคิว โดยตารางหลักในโครงสร้างข้อมูลระบบได้รับการออกแบบดังนี้:
  - 5.2.1 พจนานุกรมข้อมูลตาราง player_profiles (สถิติโปรไฟล์และการจัดอันดับ MMR)
    - จัดเก็บข้อมูลรายละเอียดเฉพาะของผู้ใช้งาน คะแนนอันดับ MMR (Matchmaking Rating) และความก้าวหน้าทางการเรียน รวมถึงประวัติแพ้ชนะ
    - [วางตารางที่ 2 พจนานุกรมข้อมูลตาราง player_profiles ตรงนี้]
  - 5.2.2 พจนานุกรมข้อมูลตาราง rooms (ห้อง Photon Session สำหรับออนไลน์เรียลไทม์)
    - จัดเก็บสถานะของห้องเล่น รหัสรหัสห้องพัก และชื่อโฮสต์ห้องประมวลผลสำหรับกลไกแบบผู้เล่นหลายคนออนไลน์
    - [วางตารางที่ 3 พจนานุกรมข้อมูลตาราง rooms ตรงนี้]
  - 5.2.3 พจนานุกรมข้อมูลตาราง matchmaking_queue (ระบบจัดการคิวเพื่อชิงสิทธิ์จับคู่เล่นอัตโนมัติ)
    - จัดเก็บข้อมูลผู้เล่นที่กดรอค้นหาเกม คัดแยกกลุ่มตามจำนวนผู้เล่นเป้าหมาย (2, 3 หรือ 4 คน) และบันทึกรหัสคำขอระบุกระบวนการเพื่อซิงค์ข้อมูลอย่างถูกต้อง
    - [วางตารางที่ 4 พจนานุกรมข้อมูลตาราง matchmaking_queue ตรงนี้]
  - 5.2.4 โครงสร้างไฟล์คลังข้อคำถามวิชาการแบบออฟไลน์ (quiz_database.json)
    - ข้อมูลคำถามเชิงการศึกษาทั้งหมดจะเก็บไว้ในรูปแบบข้อมูลโครงสร้างไฟล์ JSON ท้องถิ่น (Local Resources) ภายในตัวเกม เพื่อลดปริมาณการดึงข้อมูลเครือข่าย โดยมีรูปแบบโครงสร้าง (Schema) จัดเก็บดังนี้:
    - [วางโครงสร้างไฟล์ข้อมูลคลังคำถาม quiz_database.json ตรงนี้]

- 5.3 การออกแบบกระบวนการทำงานของระบบ (Process Design)
  - 5.3.1 แผนภาพกระแสข้อมูลของระบบ (Data Flow Diagram - DFD)
    - แผนภาพกระแสข้อมูลระดับ 0 (Context Diagram): แสดงขอบเขตการไหลเข้าออกของกระแสข้อมูลภายนอกและภายในระบบเกมการ์ด ระหว่างตัวผู้เล่นทั่วไป, บอทคอมพิวเตอร์จำลอง, ระบบลงทะเบียนทางเว็บ, ระบบประมวลผลเครือข่าย Photon Fusion และฐานข้อมูลกลาง Supabase
    
    ```mermaid
    graph TD
        %% Entities
        Player(["ผู้เล่น (Player)"])
        Bot(["บอทจำลอง (Simulated Bot)"])
        WebReg(["ระบบลงทะเบียนเว็บ (Web Registration UI)"])
        Fusion(["เซิร์ฟเวอร์ Photon Fusion 2"])
        Supabase[("ฐานข้อมูลคลาวด์ Supabase (PostgreSQL)")]

        %% Central Process
        System("1.0 ระบบเกมการ์ดวิชาการแบบออนไลน์<br>(Academic GameCard Online System)")

        %% Data Flows
        Player -->|1.1 ข้อมูลลงชื่อเข้าใช้<br>1.2 คำร้องขอเข้าคิวจับคู่<br>1.3 แอคชั่นการเล่นและคำตอบควิซ| System
        System -->|1.4 ข้อมูลโปรไฟล์และคะแนน MMR<br>1.5 คำถามควิซประจำเทิร์นและลำดับชิงสิทธิ์<br>1.6 ผลการสวมใส่กรอบโปรไฟล์| Player

        Bot -->|2.1 สุ่มความเร็วและคำตอบควิซของบอท<br>2.2 ข้อมูลการเดินหมากจำลองของบอท| System
        System -->|2.3 สถานะกระดานบอร์ดและการเรียกควิซบอท| Bot

        WebReg -->|3.1 ข้อมูลการสมัครสมาชิกใหม่| System
        System -->|3.2 ผลลัพธ์การลงทะเบียน| WebReg

        System -->|4.1 ข้อมูลกระบวนการทำแอคชั่นในเทิร์น<br>4.2 การเปิด Session ห้องออนไลน์| Fusion
        Fusion -->|4.3 สถานะเกมออนไลน์ของผู้เล่นอื่น<br>4.4 ข้อมูลซิงค์เทิร์นและตัวแปรแบบเรียลไทม์| System

        System -->|5.1 คำขอตรวจสอบบัญชีผู้ใช้<br>5.2 อัปเดต MMR, Gems และประวัติแข่งขัน| Supabase
        Supabase -->|5.3 ข้อมูลบัญชีและโปรไฟล์ที่ยืนยันแล้ว<br>5.4 ข้อมูลกระดานจัดอันดับแชมเปี้ยนบอร์ด| System
    ```

    - **คำอธิบายพจนานุกรมข้อมูลกระแสข้อมูลของแผนภาพระดับ 0 (Context Diagram Data Flows):**
      - **ข้อมูลนำเข้า (Inputs):**
        - *ข้อมูลบัญชี/การเล่นของผู้เล่น:* ผู้ใช้งานกรอกข้อมูลบัญชีเข้าใช้งาน ส่งคำสั่งขอค้นหาคู่เล่น ส่งการ์ด/เหรียญที่เลือกบนกระดาน และเลือกคำตอบควิซพร้อมบันทึกช่วงความเร็วเวลาในการตอบคำถาม
        - *ตรรกะและการตอบสนองของบอท:* บอทจำลองคำนวณการเดินหมากตามการ์ดที่มีบนกระดาน และคำนวณช่วงเวลาการตอบคำถามควิซแบบสุ่มความหน่วง 0–5 วินาทีหลังผู้เล่นตอบ
        - *การลงทะเบียนทางเว็บ:* ผู้สมัครสมาชิกกรอกรหัสและข้อมูลบัญชีเพื่อใช้สำหรับการล็อกอินครั้งแรก
        - *ระบบซิงค์เรียลไทม์ (Photon):* ส่งค่าสถานะการกระทำของผู้เล่นอื่นๆ ในกระดานรวมถึงจำนวนเหรียญและการ์ดที่เหลืออยู่
        - *ข้อมูลโปรไฟล์ที่ยืนยัน (Supabase):* ส่งค่าข้อมูลสถิติจำนวนอัญมณี (Gems), คะแนน MMR, และรายการกรอบโปรไฟล์ไอเทมที่ได้รับอนุญาตแล้ว
      - **ข้อมูลส่งออก (Outputs):**
        - *การแสดงผลข้อมูลแก่ผู้เล่น:* ระบบแสดงค่าสถิติ ความก้าวหน้าทางการเรียน แสดงหน้าจอควิซ และเรียงลำดับเทิร์นการเล่นให้ชิงสิทธิ์ตามความเร็ว
        - *ข้อมูลเซสชันและการเข้าเซิร์ฟเวอร์:* สร้างและลบคิวในฐานข้อมูล พร้อมส่ง ID ห้องให้เชื่อมต่อ Photon Fusion 2
        - *บันทึกความก้าวหน้าลงฐานข้อมูล:* อัปเดตตารางโปรไฟล์หลังจบเกม บันทึกผลลัพธ์แพ้ชนะ และหักลบจำนวน Gems ในร้านค้า

    - แผนภาพกระแสข้อมูลระดับ 1 (DFD Level 1): แสดงการแบ่งส่วนกระบวนการทำงานหลัก (Decomposed Processes) ออกเป็น 3 ส่วนการทำงานหลักอย่างชัดเจนตามสถาปัตยกรรมของเกมการ์ดวิชาการ ดังนี้:

      - **Process 1 : จัดการบัญชีผู้ใช้งานและยืนยันตัวตน (User Account & Authentication Management)**
        - รับผิดชอบกระบวนการสมัครเข้าใช้งาน ยืนยันตัวบุคคล และการปรับแต่งโปรไฟล์ โดยประสานงานร่วมกับฐานข้อมูลสมาชิก
        
        ```mermaid
        graph LR
            User(["ผู้เล่น (User)"])
            P1(("Process 1:<br>จัดการบัญชีผู้ใช้งาน<br>และยืนยันตัวตน"))
            MemberDB[("Member DB")]

            %% User to P1
            User -->|ข้อมูลการสมัครสมาชิก| P1
            User -->|ข้อมูลยืนยันตัวตน| P1
            User -->|ข้อมูลแก้ไขโปรไฟล์ / รูปภาพ| P1

            %% P1 to User
            P1 -->|สถานะการล็อกอิน| User

            %% P1 to DB
            MemberDB -->|ผลการตรวจสอบสิทธิ์| P1
            P1 -->|บันทึกสมาชิกใหม่| MemberDB
        ```

      - **Process 2 : ดำเนินการเล่นเกมและแสดงผลการ์ดความรู้ (Gameplay Execution & Knowledge Card Display)**
        - รับผิดชอบกลไกบอร์ดเกมหลัก การซื้อ/จองการ์ดคำถาม, การคำนวณและประมวลผลคำตอบควิซแทรก และการบันทึกประวัติผลแมตช์ลงแหล่งข้อมูล

      - **Process 3 : ดำเนินการทดสอบประจำวัน (Daily Knowledge Evaluation)**
        - รับผิดชอบในส่วนโหมดท้าทายประจำวันของผู้เล่นสำหรับการชิงรางวัลอัญมณีและเหรียญรางวัลพิเศษประจำวัน โดยมีโครงสร้างดังนี้

    - **คำอธิบายพจนานุกรมข้อมูลกระแสข้อมูลของแผนภาพระดับ 1 (Decomposed Process Data Flows Dictionary):**
      - **คำอธิบายในกระบวนการที่ 1 (Process 1):**
        - *ข้อมูลนำเข้าจากผู้เล่น:* ผู้เล่นกรอกฟอร์มเพื่อส่งข้อมูลบัญชีการสมัครสมาชิกใหม่, ส่งรหัสข้อมูลเพื่อตรวจยืนยันตัวตน, หรืออัปเดตรูปภาพโปรไฟล์ส่วนตัว
        - *การจัดการฐานข้อมูล:* กระบวนการจะดึงข้อมูลบัญชีจาก `Member DB` เพื่อตรวจยืนยันตัวตน เมื่อผลลัพธ์ผ่านจะส่งผลสถานะตอบกลับล็อกอินสำเร็จไปยังหน้าจอผู้เล่น หากเป็นสมาชิกใหม่ระบบจะเขียนเพิ่มบัญชีใหม่บันทึกกลับลง `Member DB`
      - **คำอธิบายในกระบวนการที่ 2 (Process 2):**
        - *ข้อมูลการกระทำบอร์ดเกม:* ผู้เล่นส่งการตัดสินใจซื้อ/จองการ์ด ส่งคำสั่งหยิบเหรียญทรัพยากรบนกระดานออนไลน์ และตอบคำถามโจทย์ควิซแทรก
        - *การประสานงานคลังความรู้:* กระบวนการดึงการ์ดโครงสร้างจาก `Card Catalog` นำส่งข้อมูลรูปและความรู้ให้ผู้เล่น ดึงคำถามสุ่มจาก `Quiz Bank` คั่นกลาง เมื่อจบเกมจะทำการบันทึกสถิติลงตาราง `Match History` และปรับอันดับอัปเดตคะแนนรวมลงใน `Member DB`
      - **คำอธิบายในกระบวนการที่ 3 (Process 3):**
        - *ข้อมูลภารกิจประจำวัน:* ผู้เล่นส่งคำขอดึงโจทย์ประจำวันและตอบตัวเลือกคำถาม โดยระบบจะส่งสถานะเพื่อเช็กสิทธิ์ล็อกอินจาก `Quiz Bank` พร้อมกับดึงโจทย์ชุดคำถาม ตัวเลือก และเฉลยออกมาตรวจเช็กคำตอบของผู้เล่น เมื่อตอบถูกจะทำการอัปเดตสถานะขีดเขียนบันทึกประวัติความก้าวหน้าลง `ประวัติการทำโจทย์` และจ่ายของรางวัลโดยสั่งอัปเดตยอดสะสม Gems/เหรียญถาวรลงใน `Member DB` ของผู้เล่นโดยตรง
  - 5.3.2 แผนภูมิขั้นตอนหลักของระบบการเล่นและการประเมินความเร็ว (Core Gameplay Flowchart)
    - แสดงโฟลว์กระบวนการทำงานในฝั่งห้องเล่นเกม เริ่มตั้งแต่การเริ่มเกมออนไลน์ -> การหมุนเทิร์นผู้เล่น -> การตัดเข้าสู่รอบตอบคำถามควิซคั่นกลาง (Interventional Quiz) ทุกๆ 3 รอบการเล่น -> การตรวจสอบเวลาตอบคำถามโดยตัวนับเวลา 10 วินาที -> กรณีผู้เล่นตอบช้าใน 5 วินาทีหลังบอทจำลองจะสุ่มการเข้าตอบและประเมินผลลัพธ์ -> การส่งสถานะผลประเมินเพื่อจัดเรียงอันดับความรวดเร็วในการชิงสิทธิ์ได้เทิร์นก่อน-หลัง -> การกระทำในเทิร์นของผู้เล่น -> การปิดหน้าตักคำนวณคะแนนเมื่อสะสมดาวครบ 20 แต้มเพื่อยุติเกม
    - [วางภาพที่ 3 แผนภาพขั้นตอนหลักของระบบการเล่นเกม Core Gameplay Flowchart ตรงนี้]
